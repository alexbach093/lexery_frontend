# Правила та Гайдлайни для ШІ-агентів (Next.js)

## Огляд

Цей документ містить суворі правила написання коду та архітектурні стандарти для ШІ-агентів та розробників у цьому Next.js проєкті. Головна мета — генерація професійного, читабельного та правильно структурованого коду.

**Головний принцип:** Код оцінюється за двома критеріями — **Функціональність** та **Структура**. Якщо код працює, але має погану структуру, він вважається сирим, незавершеним та неприйнятним.

## 1. Роутинг та директорія `src/app`

- **Чистий роутинг:** Директорія `src/app` використовується виключно для роутингу.
- **Ніяких зайвих файлів:** Тут повинні знаходитись ТІЛЬКИ сторінки та допоміжні файли Next.js (наприклад, `layout.tsx`, `loading.tsx`, `error.tsx`).
- Заборонено зберігати у `src/app` звичайні UI-компоненти, хуки чи утиліти.

## 2. Компоненти (`src/components`)

- **Розташування:** Усі компоненти мають знаходитись у `src/components`.
- **Групування:** Компоненти слід групувати у відповідні піддиректорії за їхнім контекстом чи фічею (наприклад, `src/components/chat`). Якщо підходящої директорії немає — створіть нову.
- **Назви файлів:** Назви файлів компонентів повинні бути у форматі **CamelCase / PascalCase** (наприклад, `ChatBubble.tsx`, `UserProfile.tsx`).

## 3. Іконки (`src/components/icons`)

- **Винесення іконок:** Категорично уникайте використання inline SVG безпосередньо у файлах сторінок або звичайних компонентів.
- **Формат компонента:** Усі іконки за можливості мають бути винесені у вигляді окремих React-компонентів у директорію `src/components/icons`.
- **Назви файлів:** Файли іконок повинні називатися у форматі **kebab-case-icon.tsx** (наприклад, `arrow-right-icon.tsx`, `user-avatar-icon.tsx`).
- **Групування:** Файли можна класти прямо в корінь `src/components/icons`, але за необхідності дозволяється створювати піддиректорії для їх групування.

## 4. TypeScript та Типізація

- **Типи пропсів:** Типи або інтерфейси для props компонента можуть залишатися у файлі самого компонента.
- **Глобальні типи:** Усі інші типи TypeScript (бізнес-логіка, моделі даних тощо) повинні виноситись у `src/types/index.ts`.

## 5. Стилізація

- **Тільки Tailwind CSS:** Для стилізації використовуйте виключно класи Tailwind.
- **Ніяких inline-стилів:** Атрибут `style={{}}` суворо заборонений. Його можна використовувати лише у виняткових випадках, коли стилі обчислюються динамічно через JS і без цього обійтися неможливо.

## 6. Якість та читабельність коду

- **Ліміт рядків:** Жоден згенерований файл **НЕ МОЖЕ перевищувати 1000 рядків**. Якщо файл стає завеликим, розбийте його на менші компоненти.
- **Пріоритет читабельності:** Код має бути читабельним, компактним та зрозумілим для людини. Він не повинен виглядати як машинний код.
- **Уникнення антипатернів:** Максимально уникайте будь-яких структурних антипатернів (наприклад, величезних монолітних компонентів або проп-дрилінгу).

## 7. Робота з існуючим кодом

- **Не чіпати старий код:** Наразі структура деяких старих файлів може бути не ідеальною. Під час генерації нового коду (наприклад, у новій гілці) ШІ **НЕ ПОВИНЕН** переписувати старі файли.
- **Область дії:** Застосовуйте ці правила виключно до нового коду. Змінюйте старий код тільки у випадку, якщо без цього технічно неможливо реалізувати нову функціональність.
